やること = 手元の小さなモデルにコードの置き場について質問すると、モデルが必要なファイルを自分で読みながら答える。読んだ中身はその回答の間だけ使い、次に持ち越す会話には質問と答えだけを残す。だからモデルの小さな記憶のまま、大きな置き場を相手にできる。実装の正はコード側にあり、これはそこから起こした説明。

データの流れ(決まりは図に注釈で溶かした)

図全体の前提 人が話す相手は1つだけ 人が言葉を交わすのは小さい方のモデルだけ。ファイルを読むのも、強いモデルに聞くのも、その回答を作る間だけモデルが手を伸ばす「道具」であって、人が直接触る面ではない。
① 源(これが正)
人が打つ質問
ディスク上のコードの置き場(読み取りのみ)
強い方のモデル(明示で頼まれた時だけ・任意)
↳ この会話は1つの置き場についての1本
質問が届く →
小さいモデルが答え始める▶
② 加工(仕組みの芯)
小さいモデルが答える。足りなければ「このファイルを読んで」と注文し、読み直してまた続ける。1回の質問で最大5往復まで。
決まりモデルはファイルに触らない — 文字で注文するだけ
モデルは「このパスを読んで」と決まった書式の文章を書くだけ。実際にディスクを読むのは別の読み取り専用の係で、係は置き場の外に出るパスを弾く。モデル自身はファイルにもコマンドにも一切触れない。
決まり自分で読むのは自由・強い方への相談は明示の時だけ
足りない情報は自分でどんどん読む(手元で安いので自由に)。だが強い方のモデルに回すのは、人がはっきりそう頼んだ時だけ(遅く高いので温存し、要点だけ持ち帰る)。
背景
読むのと相談するのを同じ重さにすると、小さいモデルが何でも強い方に丸投げして遅く高くなる。逆に全部自分で読ませると弱いモデルの限界に当たる。そこで「読むのは自律で安く何度でも、強い方は人が名指しした時だけ」という非対称に固定した。強い方の答えは要点だけ短く切り詰めて持ち帰る。
読んで注文して、を繰り返し
最後に文章で答える▶
③ 出力
人へ = モデルの最終回答。
画面へ = どのファイルを何バイト読んだか(名前と量だけ)。
決まり残す会話は「質問と最終回答」だけ・中身はその回限りで消える
1回の回答を作る間、読んだファイルの中身は使い捨ての作業机に積まれ、回答が終わると机ごと捨てる。次に持ち越す会話には人の質問と最終回答しか残さない。痩せているのは回をまたぐ会話の方で、1回分の作業机はその回だけ膨らむ。
背景
この道具の狙いは「小さな記憶のモデルで大きな置き場を扱う」こと。読んだファイルの中身を会話に積み続けると、数回のやり取りで小さな記憶が埋まってしまう。だから中身は回答を作る間だけの使い捨てにして、記憶に残すのは質問と答えという要点だけにした。これが道具全体が成り立つ核の決まり。
決まり画面は中身を持たない — 読んだ事実だけ
画面に流れるのは読んだファイルの名前とバイト数だけで、中身そのものは一度も画面に出さない。「誰がどのファイルを読んだか」が見えることこそがこの道具の売り。
↺ 会話が記憶の上限まで伸びる → 古いやり取りをひとまとめの要約に畳んで続ける(要約に失敗した時は畳まず、そのまま続ける)
画面に流す /
人が読み、また質問する▶
④ 受け手
人がブラウザで読む。回答と並んで「どのファイルを読んだか」がカードで見える。過去の会話は横に残り、いつでも再開できる。
決まり会話は1つの置き場に固定・過去の会話は残る
相手にする置き場は会話の頭で1つ選んで固定し、変えたい時は新しい会話を始める。会話はディスクに保存され、再起動しても横の一覧から再開できる。
図全体の総括 小さいモデルを何度も安く・強いモデルは要点だけ 小さなモデルを何度も安く回し、読んだ中身は回ごとに蒸発させる。だから小さな記憶のまま大きな置き場を扱える。強いモデルは人が名指しした時だけ呼び、その要点だけを持ち帰る。

② 加工を駆動 — 小さいモデルに毎回渡す道具の指示(全文)

1回ごとに先頭へ差し込まれ、会話には残さない。「Repo root」の行だけは今の置き場の絶対パスが埋まる。強いモデルへの相談が使える構成では、下半分の相談の段も付く。

You are the worker assistant of a cockpit over a code repository at the path given as 'Repo root'. You answer the user about this repo.
Repo root: <今の置き場の絶対パス>
TOOL fetch — read the repository yourself. Whenever answering needs to see files, output EXACTLY ONE fenced code block tagged fetch, one command per line, and nothing else:
```fetch
READ <relpath>            full file (capped)
READ <relpath> <a>-<b>    only lines a..b
GREP <regex> [reldir]     search file contents
LS [reldir]               list a directory
TREE [reldir]             tree under a directory
```
Paths are relative to the repo root. Ask only for what you need (<=8 items). The contents come back in the next message; then either fetch more or give your answer. Do not invent file contents — fetch to be sure. Skip fetching for small talk.
TOOL consult — ask the stronger reader model to study the repo, but ONLY when the user explicitly asks you to (e.g. asks you to ask ChatGPT, the reader, or a stronger model). Then output EXACTLY ONE fenced block tagged consult with one self-contained question in English, and nothing else:
```consult
<question>
```
Do not paste file contents into the question; the reader reads the repo itself. Its reply comes back as a message starting with [Reader's answer].
Emit at most one tool block per reply, and nothing else in that reply. Otherwise converse normally in the user's language.
② の枝(強い方への相談)を駆動 — 強いモデルに渡す依頼文(全文)

人が「強い方に聞いて」と頼んだ時だけ発火する。強いモデルにも同じ読み取りの書式を渡し、置き場のファイル一覧と質問を添える。本文(Repo root / File tree / Task の各行)は下の指示文の後に続けて埋まる。

You are helping me work on a code repository. You cannot touch the filesystem; a local read-only agent fetches repo contents for you on request.

When you want to inspect the repo, output EXACTLY ONE fenced code block tagged `fetch`, one command per line:
  READ <relpath>            full file (capped)
  READ <relpath> <a>-<b>    only lines a..b
  GREP <regex> [reldir]     search file contents
  LS [reldir]               list a directory
  TREE [reldir]             tree under a directory
Rules: paths are relative to the repo root and must stay inside it; ask only for what you need (<= 8 items per turn); prefer line ranges for large files. I will reply with the contents. When you are not fetching, just talk normally (no fetch block). When you have a fix, give a unified diff or the full updated file. Reason and answer in English, even if the task text below is in another language.

Repo root: <置き場のフォルダ名>

File tree:
<フィルタ済みのファイルツリー>

Task:
<人 or 小さいモデルが投げた質問>
