3レーン → 単一エージェント会話に畳むか

畳むこと自体はほぼ合意・ほぼ可逆(index.html だけ)。本当に決めるのは「観測=誰がどのファイルを読むか、を残すか捨てるか」。それがツールの identity を決める。
設計は解けた(settle 手前 ~90%)。 両フォークに号令済み → あとは実装。今出す最後の号令: この形で作ってよいか / 直す所はないか。

決着済み(再燃させない)§1 / 今セッションの流れ

✓ 人間は worker としか話さない(左の入力は削除して純ミラー化・consult も explore も worker のツールに)。UX はずっと「右1本」へ収束してきた ── 今回の1レーン案はその自然な終点。
✓ worker は2ツールのエージェント: fetch(自律・安い) / consult(明示・高い)。tool 呼び出しは既に 🔧 行に畳んで表示済み(toolcall イベント)。
✓ 単一レーン化に サーバ変更はほぼ不要。必要な SSE は全部ある(worker delta / toolcall / fetch / served / consult status＋waiting 経過 / answer / mirror)。中央・左は同じイベントの別描画にすぎない ── ほぼ index.html だけの書き換え・可逆。
✓ 3→1レーンに畳む。観測は捨てず (A) 会話に折り畳んで残す(ChatGPT の「検索しました ▸」式)。既定は綺麗な1会話、押せば「何を読んだか/ChatGPT の探索」が展開。
✓ 長い consult(~40s): 畳んでいる間は「ChatGPTに質問中… Ns」、consult カードを展開すると live mirror(今どのファイルを読んでいるか)。唯一の live 観測を温存し「固まって見える」再発を防ぐ。
✓ status は具体段階＋経過秒(「env.py を読み中」「ChatGPTに質問中 12s」)。既に持つ toolcall.summary / waiting.secs を使う。ChatGPT 風「考え中」アニメは装飾として上に乗せるだけ。
✓ 実装は index.html を単一レーンに書き換え・サーバは触らない(mirror イベントは consult 展開用に残す)。3レーン版は git に残り可逆。
✓ ask reader ▶ ボタンは残す。ただし結果は別枠 answer カードでなく 会話に inline カードで出す(生の答＋[▸探索を見る][⭢ worker に渡す])。forward はそのカード上のアクションに。直接聞く速さを温存しつつ1レーン。
✓ consult は明示のみ堅持(現状維持・実装不変)。worker が自律で ChatGPT を呼ぶのは遅い/浪費/誤判断なので不可。fetch=自律・consult=明示の非対称は UI 変更でも動かさない。

作る形(確定した設計)§2 / 実装仕様

①
単一レーン = ChatGPT 風 agent 会話
worker との1会話。あなたの発言 → status pill(考え中/env.py を読み中/ChatGPTに質問中 Ns) → worker の回答。tool 呼び出しは会話内の折り畳みカード。
✓
②
fetch カード = 「🔍 読んだ物 ▸」
worker が自律で repo を読んだら会話に1カード。既定は畳む、展開で READ/GREP した対象＋バイト数。
✓
③
consult カード = 「🔍 ChatGPT に質問 ▸」
worker 経由でも ask reader ▶ 直接でも、会話に1カード。畳んでいる間「質問中 Ns」、展開で live mirror(今どのファイル)＋[⭢ worker に渡す]。
✓
④
入力 = textarea＋2ボタン(send / ask reader ▶)
explore は既に廃止済み。カード粒度(1呼び=1カード)は実装時の私の裁量。
✓

刺してほしい(残る低確信)§3 / 実装前の最後の盲点

↓ ① 「展開カードで観測温存」まで作り込むと 3レーンの複雑さが1レーンに凝縮されるだけで、あなたが本当に欲しい「familiar でシンプル」から外れるリスク。対策 = まず既定(畳んだ)状態を綺麗に作り、展開は最小装飾で。もし作ってみて重かったら (B)→pure status に落とすのは1レーン内で安く可能。 ② 私は既存の左ミラー描画(renderMirror)を consult カードに移植できる前提。もし mirror イベントの粒度が live 表示に足りなければ、そこだけ微修正が要る(サーバ不変の前提が崩れる唯一の点) ── 実装で最初に確かめる。
